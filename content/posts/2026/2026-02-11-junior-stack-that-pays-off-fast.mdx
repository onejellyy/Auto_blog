## 스택 선택 기준은 유행보다 전환 비용입니다
주니어 시기에는 무엇을 많이 아는가보다, 배운 내용을 실제 업무에 얼마나 빨리 연결할 수 있는가가 더 중요합니다. 실무에서는 기술 난이도보다 팀의 유지보수 방식, 배포 체계, 협업 규칙에 적응하는 비용이 더 크게 작용합니다.

그래서 스택 선택은 "멋진 기술"보다 "전환 비용이 낮은 기술" 중심으로 접근하는 편이 효과적입니다.

## 바로 써먹기 좋은 기본 조합

### 프론트엔드: TypeScript + Next.js + Tailwind CSS
TypeScript는 협업 시 오류를 초기에 줄여주고, Next.js는 라우팅과 배포 구조를 빠르게 고정해줍니다. Tailwind CSS는 디자인 시스템이 없을 때도 일관된 UI를 만들기 쉬워 초기 생산성이 높습니다.

### 백엔드: Node.js + PostgreSQL + Prisma
Node.js는 프론트와 언어를 맞춰 학습 부담을 줄일 수 있습니다. PostgreSQL은 문서와 커뮤니티 자산이 풍부해 문제 해결 속도가 빠르며, Prisma는 스키마 중심 개발로 데이터 모델 변경 이력을 관리하기 쉽습니다.

### 운영: GitHub Actions + Vercel 또는 Docker 기본기
배포 자동화와 기본 CI를 먼저 익히면, 개인 프로젝트도 "운영 가능한 결과물"로 보이기 시작합니다. 이는 이력서와 포트폴리오 신뢰도를 높이는 데 직접적인 도움이 됩니다.

## 학습 순서를 잘못 잡을 때 생기는 문제
많은 학습자가 대형 아키텍처를 먼저 따라가다가 중간에 멈춥니다. 주니어 단계에서는 마이크로서비스, 복잡한 메시징 시스템보다 단일 서비스에서 인증·데이터·배포를 끝까지 연결하는 경험이 더 큰 자산이 됩니다.

실무에서는 완벽한 기술 선택보다, 제한된 시간 안에 품질 있는 완성본을 만드는 능력이 더 자주 평가됩니다.

## 요약
주니어에게 유효한 스택은 최신성보다 전환 속도와 유지보수 용이성으로 결정되는 경우가 많습니다. TypeScript, Next.js, PostgreSQL처럼 학습 자산이 크고 배포까지 연결되는 조합이 즉시 활용성이 높은 편입니다.

## FAQ
### Q. Rust나 Go를 먼저 공부하면 불리한가요?
불리하다고 단정하기는 어렵습니다. 다만 첫 포트폴리오의 완성 속도와 협업 적응 속도 측면에서는 생태계가 큰 스택이 유리하게 작동하는 경우가 많습니다.

### Q. 프레임워크보다 CS 기초가 먼저인가요?
두 축을 병행하는 편이 현실적입니다. 실무 투입 관점에서는 배포 가능한 결과물을 만들면서 자료구조, 네트워크, DB 기초를 보강하는 방식이 지속 가능성이 높습니다.
